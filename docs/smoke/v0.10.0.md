# Smoke Test — bridge v0.10.0

**Release:** [v0.10.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.10.0) · **Date:** 2026-05-16

Completes dm-assistant#506 / S10b — **faction import**, the v3 item
the #450 epic never built. Factions import as JournalEntries through
the same lifecycle as shops/locations; the Journal-tab scoped picker
now offers Faction.

**⚠️ `min-api-contract-version` 0.5.0 → 0.6.0** — faction import has
no fallback (needs `/foundry/faction/{slug}`, dm-a v0.32.0). Older
dm-a → red chip + import disabled.

Sections:

- A — Install + version chip + contract gate
- B — Faction in the Journal-tab picker
- C — Import a faction → JournalEntry
- D — Metadata header + DM/public page split
- E — Idempotency + drift
- F — Regression: shop/location/object/actor unaffected

## Prerequisites

- dm-assistant **≥ v0.32.0** on the Pi (contract 0.6.0). Hard
  requirement — see §A.
- bridge v0.10.0 in Foundry v13 + dnd5e v5.x.
- A saved faction (`dm/faction_<slug>.md`) with public sections
  (Charter / Public Goals / Membership) and DM sections (Plot Hooks /
  Suppressed Truths / DM Notes), ideally with a sigil image.

---

## A. Install + version chip + contract gate

Update from the v0.10.0 manifest URL. Reload the world.

### A — Expected

- Chip **green**, version `v0.10.0`, against dm-a ≥ v0.32.0
  (contract `0.6.0`).
- Against dm-a **< v0.32.0**: chip **red**, import disabled, hint
  says the running dm-assistant is older than the bridge minimum
  (`min-api-contract-version` is now `0.6.0`). Expected breaking-compat
  behaviour — upgrade dm-a to clear it.

---

## B. Faction in the Journal-tab picker

1. Open the **Journal** sidebar tab → **Import from dm-assistant**.

### B — Expected

- The kind toggle offers **Shop**, **Location**, **Faction** (Faction
  is new). The Actors and Items tab pickers are unchanged (no Faction
  there).
- The Faction list populates from `/faction-generate/saved`.

---

## C. Import a faction → JournalEntry

1. Journal tab → Import → **Faction** → pick your faction → **Import**.

### C — Expected

- Toast: `Imported faction <slug> — journal created.`
- A JournalEntry appears in the `<prefix> — Factions` folder, named
  after the faction, with the sigil image (if any) as its art.
- F12:

  ```js
  const j = game.journal.getName("<Faction Name>");
  console.log(
    j.getFlag("dm-assistant-bridge","kind"),   // "faction-journal"
    j.getFlag("dm-assistant-bridge","slug"),   // the faction slug
    j.pages.contents.map(p => p.name),
  );
  ```

## D. Metadata header + DM/public page split

### D — Expected

- One `JournalEntryPage` per `sections[]` entry then per
  `dm_sections[]` entry, in response order. Public pages (Charter /
  Public Goals / Membership) first, then GM-locked DM pages (Plot
  Hooks / Suppressed Truths / DM Notes).
- Page 1 has a metadata header: **Region** badge + **Members** /
  **Allied** / **Holdings** cross-links (`@UUID[...]` placeholders +
  slug-hint comments) when `related_npcs` / `related_factions` /
  `related_locations` are set in front-matter.
- The DM/public split is server-decided (`_FACTION_DM_SECTION_HEADINGS`)
  — no DM heading leaks into a public page.

---

## E. Idempotency + drift

1. Re-import the same faction.
2. Hand-edit the JournalEntry, then re-import again.

### E — Expected

- Re-import updates the existing entry in place — **no duplicate** in
  the Factions folder; pages are replaced, not appended.
- Drift policy: the dm-a-sourced entry is overwritten; a **hand-made**
  JournalEntry you create yourself (no `dm-assistant` flag) survives.

---

## F. Regression — other kinds

1. Import a Shop + Location (Journal tab), an NPC + Creature (Actors),
   an Object (Items).

### F — Expected

- All import exactly as in the bridge v0.9.x smokes. The faction
  addition is purely additive — shop/location journals, actor import,
  object Items, compendium resolution all unregressed.

## Known limits (not bugs)

- **Native JournalEntry only.** Like shop/location today, faction
  imports as a native Foundry JournalEntry. The Campaign Codex
  `group`-sheet conversion is a separate unbuilt concern (no journal
  kind does CC conversion in the bridge yet) — not part of #506.
- **No structured membership graph.** Member / allied-faction /
  holding links are `front_matter.related_*` passthrough rendered as
  `@UUID` placeholders (same as location), not a resolved graph. A
  re-resolution sweep / richer model is a follow-up.
- **Faction image is uploaded** like the shop/location hero image
  (FilePicker into `Data/<prefix>/<campaign>/faction/`), unlike the
  object remote-URL approach.
