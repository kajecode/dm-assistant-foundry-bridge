# Smoke Test — bridge v0.7.0

**Release:** [v0.7.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.7.0) · **Date:** 2026-05-15

Objects-Library round-trip (dm-assistant#502 v2a). When an imported
item's name matched a **registered campaign object** in dm-a, the
bridge builds the Item from the DM-authored object — real name, DM
lore, image — instead of an LLM stub or a same-named compendium item.

**Precedence**: `object_slug` (dm-a Objects Library, authoritative) →
`compendium_source` / name-search (#32, SRD/DDB) → LLM stub.

**Narrative-only in v2a** — the resolved object Item has no structured
mechanics (like the v0.5.2 spell stubs); the GM/compendium refines.

Sections:

- A — Install + version chip + the contract-bump gate
- B — Homebrew object resolves to the DM-authored object
- C — SRD item on the same NPC still name-searches (#32 unregressed)
- D — Un-registered invented item still stubs
- E — Drift policy + no library-folder copy for object items
- F — Object endpoint down → graceful fallback

## Prerequisites

- **dm-assistant ≥ v0.30.0** on the Pi (ships `/foundry/object/{slug}`
  + populates `object_slug`; API contract **0.5.0**). This is a hard
  requirement — see section A.
- bridge v0.7.0 in Foundry, Foundry v13 + dnd5e v5.x.
- A campaign with **a registered Objects-Library object** with a
  distinctive name and ideally a generated image — e.g. **Thorncall
  Blade** (slug `thorncall-blade`).
- An NPC whose Stats Block actions reference that object **by exact
  name** ("Thorncall Blade"), plus at least one SRD item (e.g.
  "Longsword" or "Dagger") and one invented one-off. Re-roll the
  Stats Block + re-extract if needed so the names land verbatim.
- For section C: the `itemCompendiums` setting configured (`auto` or
  the SRD pack ids) — same as the v0.6.0 smoke.

## B. Homebrew object resolves to the DM-authored object

1. Confirm the object is registered in dm-a (Objects panel) — note
   its slug (`thorncall-blade`) and that it has DM lore + an image.
2. Import the NPC that wields it.

### B — Expected

- The actor has an Item named **`Thorncall Blade`** — the object's
  **authored name**, NOT the `(NPC Name)`-decorated stub form.
- Its description is the **DM-authored object lore** (rendered HTML),
  not the LLM's terse stub line.
- If the object had an image, the Item art points at the dm-a object
  image URL.
- F12 check:

  ```js
  const it = game.actors.getName("<NPC>").items.find(i => i.name === "Thorncall Blade");
  console.log(
    it.getFlag("dm-assistant-bridge","source"),        // "dm-assistant"
    it.getFlag("dm-assistant-bridge","object_slug"),   // "thorncall-blade"
    it.getFlag("dm-assistant-bridge","resolved_from"), // "dm-assistant:object/thorncall-blade"
    it.type,                                            // best-effort item_type (e.g. "weapon")
  );
  ```
  Expect: `source: "dm-assistant"`, `object_slug: "thorncall-blade"`,
  `resolved_from: "dm-assistant:object/thorncall-blade"` (a namespaced
  provenance string, **not** a Foundry UUID), and a sensible `type`.

### B — Narrative-only caveat

The item has **no working attack/damage roll** in v2a — it's a
narrative item (real identity + lore + art). That's by design;
mechanics are v2b. The GM can flesh it out or drag a compendium
equivalent. Confirm it at least *exists* and renders cleanly on the
sheet.

## F. Object endpoint down → graceful fallback

Goal: the import never fails on an object-fetch problem.

1. Temporarily make the object un-fetchable — e.g. rename/unregister
   the object in dm-a so `/foundry/object/thorncall-blade` 404s, OR
   point the bridge baseUrl at a momentarily-stopped dm-a.
2. Re-import the NPC.

### F — Expected

- Import **succeeds**. The "Thorncall Blade" item falls through:
  - If a same-named compendium item exists + packs configured → it
    resolves via name-search (#32).
  - Else → plain LLM stub.
- F12 console shows a single `debug`-level
  `object_slug "…" did not resolve; falling back` line — no error,
  no failed import.
- Restore the object, re-import → back to the section-B result.

## Known limits (not bugs)

- **Narrative-only (v2a).** Object items have no damage/attack. v2b
  (a stats sidecar) is spec'd in dm-assistant#502 and deferred. Don't
  file "object item has no attack roll" — that's the design.
- **Object image is a remote dm-a URL.** Unlike NPC portraits (which
  are downloaded + uploaded into Foundry's Data dir), the object
  Item's `img` points at the dm-a image endpoint directly. It needs
  the dm-a server reachable from the browser viewing the sheet. A
  later release may localise it; for v2a it's a deliberate
  lightweight choice (the resolver stays self-contained + never
  throws).
- **`compendium_source` still not populated upstream — and won't be.**
  dm-a structurally cannot emit Foundry compendium UUIDs (the #502
  architectural finding). The path stays wired + unit-tested but is
  dead in practice; the SRD path is name-search, the homebrew path
  is `object_slug`.
- **Exact name match only** (object link is server-side, case-
  insensitive, against dm-a's own registry). SRD name-search is still
  exact-only (v0.6.0 limit unchanged).

## If something fails

- Capture: dm-a version (`/api/health`) + contract
  (`/api/foundry/health`), the object's `/foundry/object/<slug>`
  JSON, F12 console during import, the resolved item's `toObject()`
  (`actor.items.get(id).toObject()`), the `itemCompendiums` setting.
- File against `kajecode/dm-assistant-foundry-bridge` with the
  v0.7.0 tag.
