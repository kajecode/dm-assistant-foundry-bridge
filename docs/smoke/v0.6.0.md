# Smoke Test — bridge v0.6.0

**Release:** [v0.6.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.6.0) · **Date:** 2026-05-15

Compendium-source resolution (#32). Imported items / spells get
swapped for fully-statted compendium documents when a match exists.
**Opt-in** — no behaviour change until you set `itemCompendiums`.

Sections:

- A — Install + version chip
- B — Opt-in OFF: items stay stubs (no regression)
- C — Configure + resolve against the dnd5e SRD
- D — Library folder + drift policy
- E — No-match fallback + type-guard
- F — Regression: weapons/feats unaffected when no match

## Prerequisites

- dm-assistant ≥ 0.29.4 on the Pi (no contract change — still 0.4.0).
- bridge v0.6.0 in Foundry, Foundry v13 + dnd5e v5.x.
- An item compendium installed. Easiest: the **dnd5e SRD** pack
  (`dnd5e.items`, `dnd5e.spells`) that ships with the system. If you
  run **DDB Importer**, its packs work too.
- A spellcaster NPC whose actions sidecar has named spells (Elowen,
  post-v0.29.4 re-roll + re-extract).

## A. Install + version chip

Update the module from the v0.6.0 manifest URL. Reload the world.

### A — Expected

- Chip green, version `v0.6.0`, contract `0.4.0` (unchanged).

## B. Opt-in OFF — no regression

Goal: confirm the default (empty setting) leaves v0.5.2 behaviour
exactly as-is.

1. **Game Settings → Module Settings → DM Assistant Bridge.** The new
   **"Item compendiums to resolve against"** field is present and
   **empty**.
2. Import Elowen (or re-import).

### B — Expected

- Spells / weapons / feats appear exactly as in the v0.5.2 smoke —
  stub items (spells level-0/cantrip, empty school).
- No `<prefix> — Items` folder created.
- F12: `game.actors.getName("Elowen Tristane").items.find(i=>i.type==="spell").getFlag("dm-assistant-bridge","resolved_from")` → `undefined` (still a stub).

---

## C. Configure + resolve

1. Set the **Item compendiums** setting to `auto` (search every Item
   pack) — or list specific ids, e.g. `dnd5e.items, dnd5e.spells`.
2. Save settings.
3. Re-import Elowen.

### C — Expected

- Spells whose names match the SRD (Druidcraft, Fireball, Cure
  Wounds, Barkskin, Moonbeam, …) are now **fully-statted spell
  items**: real level (Moonbeam = 2, Fireball = 3), school,
  components, range, damage — not level-0 stubs.
- Item names are the **catalogue names** (`Moonbeam`), NOT the
  `(Elowen Tristane)`-decorated stub form.
- F12 check on a resolved spell:

  ```js
  const s = game.actors.getName("Elowen Tristane").items.find(i => i.name === "Moonbeam");
  console.log(s.system.level, s.getFlag("dm-assistant-bridge","source"),
              s.getFlag("dm-assistant-bridge","resolved_from"),
              s.getFlag("core","sourceId"));
  ```
  Expect: real level (2), `source: "dm-assistant"`, a `resolved_from`
  UUID, and `core.sourceId` set to the same compendium UUID (the
  sheet shows "from compendium").

### C — Spell that has NO SRD match

A homebrew / oddly-named spell with no compendium entry stays a
stub (level-0). That's correct — it falls back, doesn't error.

## D. Library folder + drift policy

### D1. Library copy

After section C, check the **Items** sidebar tab.

- A `<prefix> — Items` folder exists, containing one copy of each
  resolved compendium item (Moonbeam, Fireball, …).
- Re-importing Elowen again does **not** duplicate them (idempotent
  on the compendium UUID).

### D2. Drift policy intact

1. On Elowen's sheet, manually add an item (e.g. drag a different
   spell from the SRD, or hand-create a Feature).
2. Re-import Elowen.

#### D2 — Expected

- The dm-assistant-resolved items are dropped + recreated (drift
  policy). Your hand-added item **survives** (no `dm-assistant`
  source flag).
- The `<prefix> — Items` library folder is untouched by the
  drop-and-replace (it only cleans actor-embedded items).

## E. No-match fallback + type guard

### E1. No-match

Import an NPC with an item the LLM invented that isn't in any
compendium (e.g. a unique named weapon "Thorncall Blade").

- Expected: stays a stub. No error. Imports fine.

### E2. Type guard

If the LLM ever tags a class/background name as `loot` (e.g. an
item literally named "Fighter"), and a `class`-type compendium doc
named "Fighter" exists:

- Expected: the resolver **rejects** the `class` doc (not a bridge
  item type) and keeps the stub. No "Fighter class" document
  embedded on the actor.

(This is hard to trigger by hand — it's covered by the unit test
`rejects a compendium doc whose type isn't a bridge item type`.
Spot-check only if you happen to hit it.)

## F. Regression — weapons + feats

1. On the resolved Elowen, check the Dagger (weapon) and feats.

### F — Expected

## Known limits (not bugs)

- **Exact name match only.** "Healing Potion" resolves; "Potion of
  Greater Healing" needs that exact compendium name. Fuzzy matching
  is a documented follow-up — exact-only avoids resolving the wrong
  item (Dagger vs Dagger +1).
- **`compendium_source` not populated upstream yet.** dm-assistant
  reserves the field; #481 v2 will set it. Until then resolution is
  name-search only. The precedence path is wired + unit-tested for
  when it lands.
- **Library copy is best-effort.** If `Item.create` for the
  `<prefix> — Items` copy fails, it's logged and skipped — the
  actor still gets the resolved embedded item. The folder copy is a
  convenience library, not load-bearing.

## If something fails

- Capture: the `itemCompendiums` setting value, F12 console during
  import, the resolved item's `system` + flags
  (`actor.items.get(id).toObject()`), which compendium packs are
  installed (`[...game.packs].map(p=>p.metadata.id)`).
- File against `kajecode/dm-assistant-foundry-bridge` with the
  v0.6.0 tag.
