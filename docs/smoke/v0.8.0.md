# Smoke Test — bridge v0.8.0

**Release:** [v0.8.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.8.0) · **Date:** 2026-05-16

Objects become first-class Foundry Items (dm-assistant#504) and the
import button is split per Foundry sidebar tab, scoped (dm-assistant#505).

**No API contract change** — uses endpoints that already shipped
(`/foundry/object/{slug}` from contract 0.5.0, pre-existing
`/object-generate/saved`). Still needs dm-assistant ≥ v0.30.0.

Sections:

- A — Install + version chip (contract unchanged)
- B — Per-tab buttons appear, each scoped
- C — Standalone Object import → world Item
- D — NPC wielding a registered object → embedded + world Item
- E — Idempotency + drift on re-import
- F — Scope isolation + Lore/Faction deferral
- G — Regression: NPC/Creature/Shop/Location unaffected

## Prerequisites

- dm-assistant **≥ v0.30.0** on the Pi (contract 0.5.0). No re-deploy
  needed if v0.30.0 is already running from the prior smoke.
- bridge v0.8.0 in Foundry, Foundry v13 + dnd5e v5.x.
- A registered Objects-Library object with a distinctive name (e.g.
  **Thorncall Blade**, slug `thorncall-blade`), ideally with an image.
- An NPC whose Stats Block actions name that object verbatim (for §D —
  same NPC as the v0.7.0 smoke works).

## B. Per-tab buttons appear, each scoped

1. Open the **Actors** sidebar tab.
2. Open the **Items** sidebar tab.
3. Open the **Journal** sidebar tab.

### B — Expected

- An **"Import from dm-assistant"** button is in the directory header
  of **all three** tabs (previously: Actors only).
- Click each; the picker's kind toggle is scoped:
  - **Actors** → only **NPC**, **Creature**.
  - **Items** → only **Object**.
  - **Journal** → only **Shop**, **Location**. (No Lore/Faction yet —
    see §F.)
- No tab offers a kind that doesn't belong to it (e.g. the Items
  picker shows no NPC option).

## C. Standalone Object import → world Item

1. **Items** tab → **Import from dm-assistant** → pick **Thorncall
   Blade** → **Import**.

### C — Expected

- Toast: `Imported object thorncall-blade — Item created.`
- A new **Item** appears in the Items sidebar, inside the
  `<prefix> — Items` folder, named **Thorncall Blade** (the authored
  name), with the DM-authored lore as its description and the object
  image (if any) as its art.
- F12 check:

  ```js
  const it = game.items.getName("Thorncall Blade");
  console.log(
    it.getFlag("dm-assistant-bridge","source"),        // "dm-assistant"
    it.getFlag("dm-assistant-bridge","kind"),           // "object-item"
    it.getFlag("dm-assistant-bridge","object_slug"),    // "thorncall-blade"
    it.getFlag("dm-assistant-bridge","resolved_from"),  // "dm-assistant:object/thorncall-blade"
    it.type,                                            // best-effort item_type
  );
  ```

### C — Narrative-only caveat

Still v2a narrative-only — no working attack/damage roll on the Item
(that's v2b, dm-assistant#502). Confirm it exists, is in the right
folder, and renders cleanly.

## D. NPC wielding the object → embedded AND world Item

1. **Actors** tab → import the NPC that wields Thorncall Blade.

### D — Expected

- The actor has **Thorncall Blade embedded** (unchanged from v0.7.0 —
  mechanics need it embedded).
- The **same object also appears as the world Item** in
  `<prefix> — Items` (refreshed alongside the embed) — one Item, even
  if multiple NPCs reference it.
- Importing the standalone object (§C) and the NPC (§D) in either
  order yields exactly **one** `Thorncall Blade` world Item, not two.

## E. Idempotency + drift

1. Re-import the object (§C) a second time.
2. On the world Item's sheet, then re-import the NPC (§D).

### E — Expected

- Re-import updates the existing world Item in place — **no duplicate**
  `Thorncall Blade` in the Items folder.
- Drift policy holds: the dm-a-sourced world Item is overwritten by
  re-import; a **hand-created** Item you add yourself (no
  `dm-assistant` source flag) is left untouched.

## Known limits (not bugs)

- **Narrative-only objects.** No mechanics on the Item (v2b —
  dm-assistant#502).
- **Object image is a remote dm-a URL** (not uploaded into Foundry's
  Data dir) — unchanged from v0.7.0; needs dm-a reachable from the
  browser viewing the sheet.
- **Lore / Faction → Journal not yet built.** Need new
  `/foundry/lore|faction` endpoint families — separate dm-a + bridge
  tickets (dm-assistant#505 Out of Scope).
- **Every object-resolved stub also spawns a world Item.** Generic
  one-off stub weapons/feats are intentionally NOT mirrored to the
  Items tab in this release (object-linked items only) — keeps the
  library uncluttered. (If you want stub mirroring too, that's a
  follow-up decision.)
