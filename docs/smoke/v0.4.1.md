# Smoke Test — bridge v0.4.1

**Release:** [v0.4.1](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.4.1) · **Date:** 2026-05-14

Patch on the v0.4.0 surface. Adds a campaign-picker dropdown to the
Settings panel sourced from dm-assistant's `GET /campaigns` endpoint
(#12). No new import flow, no contract changes — purely a settings UX
polish.

Sections:

- A — Install + version chip
- B — Picker happy path: dropdown loads + persists
- C — Refresh button + stale-id handling
- D — Fallback path: free-text input still works when /campaigns is unreachable

## Prerequisites

- dm-assistant ≥ 0.25.0 reachable from Foundry (same as v0.4.0).
- At least two campaigns on the dm-assistant server so the dropdown
  has something to scroll through. If you only have one, the dropdown
  still works — just sparser.

---

## A. Install + version chip

1. Foundry → **Game Settings → Manage Modules → Update Module** for the
   DM Assistant Bridge (or install via manifest URL):

   ```text
   https://github.com/kajecode/dm-assistant-foundry-bridge/releases/latest/download/module.json
   ```

2. Reload the world.

### A — Expected

- Chip is **green**.
- Tooltip shows **bridge `v0.4.1`** (was `v0.4.0`).
- Contract version unchanged at **`0.3.0`** (no contract bump).

---

## B. Picker happy path

Goal: open the settings panel and confirm the campaign picker mounted.

1. **Game Settings → Module Settings → DM Assistant Bridge**.
2. Locate the **Campaign ID** row.

### B — Expected

- Where the v0.4.0 text input was, there's now a **dropdown** + a
  small **↻ refresh** button.
- Dropdown is sorted by campaign name (not id). Each option reads
  `<id> — <name> (<game_system>)`.
- A status line below reads `N campaign(s) available · refresh after changes`.
- The original text input is **hidden** (not removed) — you can verify
  via the browser DevTools that it's still in the form with
  `display:none`. This keeps Foundry's form-save flow intact.
- If you had a campaign id saved already, that id is **pre-selected**
  in the dropdown.

### B — Save flow

1. Pick a campaign from the dropdown.
2. Click **Save Changes**.
3. Re-open the settings panel.

#### B — Expected

- The dropdown re-mounts with the newly-saved campaign pre-selected.
- All bridge operations (Test Connection, picker import, etc.) use the
  newly-picked campaign id with no further intervention.

## C. Refresh + stale-id

### C1. Refresh button

1. Open the settings panel.
2. In a separate browser tab, create a new campaign on the dm-assistant
   side (or have a friend do it).
3. Back in Foundry settings → click the **↻ refresh** button next to
   the dropdown.

#### C1 — Expected

- Dropdown briefly disabled with `Loading…` placeholder.
- After ~1 sec: new dropdown options include the just-created
  campaign.
- Status line updates to the new count.

### C2. Stale-id surfacing

Simulate a stale id (a previously-saved campaign that was deleted on
the server side):

1. Open the settings panel. Note the current campaign id.
2. In a separate tab on dm-assistant, delete a campaign you have set
   (or rename one — same effect for this test).
3. Reload the Foundry settings panel.

#### C2 — Expected

- Dropdown shows the now-stale id at the bottom as a distinct option:
  `<stale-id> — (not in server list)`.
- The stale option is pre-selected (so the operator can see what's
  configured and choose to re-pick).
- Picking any valid option and saving recovers cleanly.

## D. Fallback path

Goal: prove that an unreachable dm-assistant doesn't break the
settings panel. The fallback should drop back to the v0.4.0 text-input
behaviour with an inline hint explaining why.

1. In settings, **temporarily** change the **dm-assistant base URL**
   to something unreachable (e.g. `http://localhost:9999`).
2. Save, then re-open settings.

### D — Expected

- The dropdown briefly tries to load, then hides itself.
- The original text input reappears (the same one that's hidden in the
  happy path).
- An inline hint below reads `Couldn't reach dm-assistant — check the URL above and that the server is running. Falling back to free-text input.`
- The text input still trims on read (the v0.1.0 defence is preserved).

### D2. HTTP-error variant

1. Set the base URL to a URL that returns 500 (or just point at a host
   you know returns garbage for `/campaigns`).
2. Re-open settings.

#### D2 — Expected

- Same fallback, but the hint message includes the **status code**:
  `dm-assistant returned 500. Falling back to free-text input.`

### D3. baseUrl unset

1. Clear the base URL field (or open settings in a fresh Foundry world
   before configuring the bridge).
2. Look at the Campaign ID row.

#### D3 — Expected

- No dropdown rendered (no point — we don't know which server to ask).
- Hint reads `Configure the dm-assistant URL above first, then re-open this panel to populate the dropdown.`
- Text input still works — you can hand-type a campaign id if needed.

### D4. Restore + re-verify

1. Set the base URL back to the working dm-assistant.
2. Re-open settings.

#### D4 — Expected

- Dropdown is back, populated with the real campaign list.
- No leftover error hint.

## Known limits (not bugs)

- **Dropdown fetches on every settings-panel render.** Not cached
  across renders. Cheap and ensures freshness; if you have hundreds of
  campaigns this might be worth caching, but at v1 that's not a real
  scenario.
- **No CORS preflight workaround** — if `/campaigns` 200s in a curl
  but 404s in the browser, it's almost certainly CORS. Add your
  Foundry origin to dm-assistant's `ALLOWED_ORIGINS` env var.

## If something fails

- Capture: screenshot of the settings panel, browser console (F12)
  output, browser network panel showing the `/campaigns` request.
- File a bug in `kajecode/dm-assistant-foundry-bridge` with the v0.4.1
  tag.
