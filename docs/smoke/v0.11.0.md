# Smoke Test — bridge v0.11.0

**Release:** [v0.11.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.11.0) · **Date:** 2026-05-17

Completes dm-assistant#507 — **lore import**, the last #450
close-blocker besides #502. Lore imports as a **player-readable**
JournalEntry (the exception to the GM-locked journal pattern). The
Journal-tab scoped picker now offers Lore.

**⚠️ `min-api-contract-version` 0.6.0 → 0.7.0** — lore import has no
fallback (needs `/foundry/lore/{slug}`, dm-a v0.33.0). Older dm-a →
red chip + import disabled.

Sections:

- A — Install + version chip + contract gate
- B — Lore in the Journal-tab picker
- C — Import lore → PLAYER-READABLE JournalEntry
- D — Pages + no metadata header + no DM split
- E — Idempotency + drift; GM-lock regression
- F — Regression: shop/location/faction/object/actor unaffected

## Prerequisites

- dm-assistant **≥ v0.33.0** on the Pi (contract 0.7.0). Hard
  requirement — see §A.
- bridge v0.11.0 in Foundry v13 + dnd5e v5.x.
- A disseminated lore doc (`shared/lore_<slug>.md`); ideally one with
  a `## Plot Hooks` or `## Secrets` heading to prove lore does NOT
  DM-split.
- A logged-in **non-GM player** (or a player-perspective check) to
  confirm visibility in §C.

## A. Install + version chip + contract gate

Update from the v0.11.0 manifest URL. Reload the world.

### A — Expected

- Chip **green**, `v0.11.0`, against dm-a ≥ v0.33.0 (contract `0.7.0`).
- Against dm-a **< v0.33.0**: chip **red**, import disabled (min
  `0.7.0`). Expected — upgrade dm-a to clear.

## B. Lore in the Journal-tab picker

1. **Journal** sidebar tab → **Import from dm-assistant**.

### B — Expected

- Kind toggle offers **Shop**, **Location**, **Faction**, **Lore**
  (Lore is new). Actors/Items tab pickers unchanged.
- Lore list populates from `/lore-generate/saved`.

## C. Import lore → player-readable JournalEntry

1. Journal tab → Import → **Lore** → pick a lore doc → **Import**.

### C — Expected

- Toast: `Imported lore <slug> — journal created.`
- A JournalEntry appears in the **`<prefix> — Lore`** folder.
- **Players can see it.** F12:

  ```js
  const j = game.journal.getName("<Lore Name>");
  console.log(
    j.ownership.default,                              // 2 (OBSERVER) — NOT 0
    j.getFlag("dm-assistant-bridge","kind"),          // "lore-journal"
  );
  ```
  Or: log in as a non-GM player → the lore journal is visible/readable.
  (Shop/Location/Faction journals remain GM-only — `ownership.default
  === 0`.)

## D. Pages + no metadata header + no DM split

### D — Expected

- One `JournalEntryPage` per `sections[]` entry, in order. **Every**
  section is a public page — even a `## Plot Hooks` / `## Secrets`
  heading (lore has no DM split; the page is NOT GM-locked).
- **No metadata header** on Page 1 (no Region/owner/related badges,
  no `@UUID[...]` — lore is pure reference prose).
- No imageless-related errors (lore has no hero image; the import
  skips the upload step cleanly).

## E. Idempotency + drift; GM-lock regression

1. Re-import the same lore doc.
2. Hand-edit the JournalEntry, then re-import again.
3. Re-import a **Faction** (from v0.10.0) in the same pass.

### E — Expected

- Re-import updates in place — no duplicate in `<prefix> — Lore`;
  pages replaced, not appended.
- Drift: dm-a-sourced entry overwritten; a hand-made JournalEntry
  (no `dm-assistant` flag) survives.
- **Regression:** the re-imported Faction journal is still
  **GM-only** (`ownership.default === 0`) — the player-readable
  treatment is lore-only and didn't leak to other kinds.

## Known limits (not bugs)

- **Lore is the one player-visible journal kind.** This is by design
  (lore = player-facing world reference). If you do NOT want players
  to see imported lore, that's a future per-import toggle — not v1.
- **Native JournalEntry only.** No Campaign Codex sheet conversion
  for lore (CC's types are entity-ish; lore is reference text) — and
  no CC conversion for any journal kind in the bridge yet.
- **Imageless.** Lore has no hero image by design.

## If something fails

- Capture: dm-a version + contract, the `/foundry/lore/<slug>` JSON,
  F12 console during import, the JournalEntry `toObject()` (esp.
  `ownership`), whether the source lore was in `shared/` vs `dm/`.
- File against `kajecode/dm-assistant-foundry-bridge` with the
  v0.11.0 tag.
