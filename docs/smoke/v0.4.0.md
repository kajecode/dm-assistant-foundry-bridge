# Smoke Test — bridge v0.4.0

**Release:** [v0.4.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.4.0) · **Date:** 2026-05-13

What this release adds (run through in order; later sections depend on earlier
gates):

- A — Install + version chip
- B — Picker dialog (now 4 kinds)
- C — Shop import end-to-end (#25)
- D — Location import end-to-end (#26)
- E — Targeted 401 hints (#28, optional — only if `FOUNDRY_API_KEY` is set on dm-assistant)
- F — Regression check: existing NPC + Creature import flows still work

## Prerequisites

- **dm-assistant ≥ 0.25.0** (contract 0.3.0) running and reachable from Foundry.
  Older deployments will see the chip go **yellow** ("outdated") and import will
  be disabled — this is by design (the breaking `min-api-contract-version` bump
  from this release).
- A campaign with at least one saved shop and at least one saved location
  **on disk as `dm/shop_*.md` and `dm/location_*.md`**. Legacy corpora may
  have shops misclassified as locations (the bridge will 404 on
  `GET /foundry/shop/{slug}` because the file is named `dm/location_*.md`).
  **If you hit a 404 in section C below, run the dm-assistant v0.26.0 corpus
  migration first** — see
  [`dm-assistant docs/smoke/v0.26.0.md`](https://github.com/kajecode/dm-assistant/blob/main/docs/smoke/v0.26.0.md)
  sections B + C. The migration ships in dm-assistant ≥ 0.26.0.
- Foundry v13. (v14 is out of the compat window.)

## A. Install + version chip

Goal: confirm the new module loads and reports itself as v0.4.0 against
a compatible dm-assistant.

1. In Foundry → **Game Settings → Manage Modules → Install Module → Manifest URL**:

   ```text
   https://github.com/kajecode/dm-assistant-foundry-bridge/releases/latest/download/module.json
   ```

   (Or upgrade the existing install if v0.3.x is already there.)
2. Enable the module and reload the world.
3. Open the bridge sidebar/header chip.

### A — Expected

- Chip is **green**.
- Tooltip shows **bridge `v0.4.0`** and the contract version
  reported by dm-assistant (should be **`0.3.0`** or higher).
- No "outdated" yellow state. No 401 red state.

### A — Gate

Stop here if the chip isn't green — every following section depends on a
working connection.

## B. Picker dialog

Goal: confirm the import picker exposes all four kinds and all four lists
populate.

1. Click the bridge's **Import** button to open the picker.

### B — Expected

- Kind toggle shows **four** radios: `NPCs`, `Creatures`, `Shops`, `Locations`.
- All four lists fetch concurrently when the dialog opens — switching radios
  is instant (no per-tab spinner).
- Each list shows entities for the current campaign, with name + slug.
- If one of the four list endpoints fails (rare), the picker still opens with
  the others populated; the failing kind's row shows an inline error rather
  than killing the whole dialog. (This is intentional `Promise.allSettled`
  behaviour.)

## C. Shop import (#25)

Goal: import a shop end-to-end, verify journal layout, folder placement, and
metadata header.

1. In the picker, select **Shops**, pick an entry with a known proprietor (e.g.
   one that was saved via `/shop-generate/save` so the registry has the link).
2. Click **Import**.
3. After the import toast, open the journal sidebar.

### C — Expected: folder + identity

- Journal lands in folder **`DM Assistant — Shops`** (or whatever prefix the
  Folder setting is set to + `Shops`). Folder is created if absent.
- Journal name matches the shop's `display_name` or `name` field.
- Journal flags (right-click → **Edit** → **F12 inspect**, or via the bridge's
  flag inspector if available) show:
  - `flags["dm-assistant-bridge"].kind === "shop-journal"`
  - `flags["dm-assistant-bridge"].slug === <shop slug>`
  - `flags["dm-assistant-bridge"].api_contract_version === "0.3.0"`
  - `flags["dm-assistant-bridge"].source_path` ends with `shop_<slug>.md`

### C — Expected: pages

- Public-section pages come first (e.g. `Shop Name & Type`,
  `Atmosphere & Description`, `Inventory`, `Special & Rare Items`,
  `Rumours the Proprietor Knows`).
- DM-section pages come last (e.g. `Shop's Secret`,
  `Proprietor: Motivation & Secret`, `Proprietor: Plot Hooks`,
  `Proprietor: Stats Block`).
- DM-section pages have **ownership locked to GM only** — when impersonating
  a player (or testing with a second logged-in player), the DM pages should
  be hidden while the public ones are readable.

### C — Expected: metadata header (Page 1 only)

- Page 1's body starts with a header paragraph rendering:
  - `Type: <shop_type>` (from front-matter; e.g. "blacksmith")
  - `Region: <region>`
  - `Proprietor: @UUID[Actor.<proprietor-slug>] <!-- bridge: actor pending; slug=<x> -->`
    (the `@UUID` will render in Foundry as a literal placeholder until the
    proprietor actor is imported separately — this is the forward-compat
    cross-link).
- Pages 2+ do **not** repeat the metadata header.

### C — Expected: hero image

- Journal's `img` is the establishment image uploaded from
  `establishment_image_url`. If the source shop has no image, the journal
  uses Foundry's default book icon (`icons/svg/book.svg`).

### C — Idempotency

- Re-run the import for the same shop. Journal is **updated in place** (not
  duplicated). Flags drive the drift-track via `kind: "shop-journal"` +
  `slug` lookup.

## D. Location import (#26)

Goal: import a location end-to-end, verify the same skeleton as C with
location-specific divergences.

1. In the picker, select **Locations**, pick an entry.
2. Click **Import**.

### D — Expected: folder + identity

- Journal lands in **`DM Assistant — Locations`**.
- Flags carry `kind: "location-journal"` (distinct from shop-journal, so a
  location + shop sharing a slug don't collide).

### D — Expected: metadata header (Page 1 only)

- `Region: <region>`
- `Area: <area>` (if present in front-matter)
- `NPCs:` line with `@UUID[Actor.<slug>]` placeholders for each
  `related_npcs` entry.
- `Shops:` line with `@UUID[JournalEntry.<slug>]` placeholders for each
  `related_shops` entry.
- `Locations:` line with `@UUID[JournalEntry.<slug>]` placeholders for each
  `related_locations` entry.
- Empty `related_*` arrays produce **no** line for that kind (not "NPCs: " with
  nothing after it).

### D — Expected: pages

- Public sections first (e.g. `Overview`, `Layout`, `Inhabitants`).
- DM sections last (e.g. `Secrets & Hidden Features`, `Adventure Hooks`).
  GM-only ownership on the DM pages.

### D — Expected: hero image

- `img` is the location's `map_image_url` upload (different field from shop's
  `establishment_image_url` — verifies the per-kind discriminated-union branch).

## E. Targeted 401 hints (#28) — optional

Only run this if you have `FOUNDRY_API_KEY` configured on dm-assistant.
Skip otherwise — the path will just stay green.

### E1. Missing key

1. In Foundry → bridge module settings → clear the **API Key** field.
2. Click **Test Connection** (or trigger any import).

E1 — Expected:

- Status copy reads **"API key required"** (not the generic
  "Server demanded authentication").
- Detail copy mentions the module-settings field by name (verbatim from the
  `hint` returned by dm-assistant's structured 401 body).

### E2. Wrong key

1. Set the bridge's **API Key** field to a deliberately-wrong value
   (`bad-key-test`).
2. Click **Test Connection**.

E2 — Expected:

- Status copy reads **"API key mismatch"**.
- Detail copy mentions **typo / rotation / whitespace** as common causes.

### E3. Restore + verify

1. Restore the correct API key.
2. Click **Test Connection**.

E3 — Expected:

- Chip returns to **green**; no 401 hint state.

### E4. Backwards-compat (older dm-assistant)

Skip unless you have a non-upgraded dm-assistant handy. If you do:

- A 401 from an older dm-assistant (pre-#489) should still produce the
  generic v0.3.x copy ("Server demanded authentication"), not crash.

## F. Regression check — NPC + Creature import still work

The shop + location work refactored `flagKindFor()` and widened `FlagKind`.
Verify the existing flows didn't drift:

1. In the picker, select **NPCs**, pick one, import.
   - Journal/actor still lands in **`DM Assistant — NPCs`**.
   - Flags show `kind: "npc-actor"`.
   - DM-notes journal (if applicable to your render path) shows
     `kind: "npc-dm-notes"`.
2. Same for **Creatures**:
   - Folder: **`DM Assistant — Creatures`**.
   - Flags: `kind: "creature-actor"` / `kind: "creature-dm-notes"`.

### F — Gate

If NPC or Creature import broke, the FlagKind widening or folder helper
refactor regressed. File a bug before considering the smoke green.

## Go / no-go gates

For this smoke to be considered green:

- [ ] **A** chip green, version reports `v0.4.0`
- [ ] **B** picker shows 4 kinds, all 4 lists populate
- [ ] **C** shop import lands in `Shops` folder with correct page order, GM-only DM pages, metadata header with proprietor @UUID placeholder, idempotent re-import
- [ ] **D** location import lands in `Locations` folder with related-entity @UUID links, region+area badges
- [ ] **E1-E3** 401 hints fire correctly (skip if no API key configured)
- [ ] **F** NPC + Creature import unaffected

## Known limits (not bugs)

- **@UUID placeholders are literal text in v1**, not live cross-links — Foundry
  renders them as `@UUID[Actor.aldric-harwick]` source. A future bridge release
  will run a resolution sweep after import to swap placeholders for real
  document IDs. The `<!-- bridge: actor pending; slug=<x> -->` HTML comment
  marks each one for that future sweep.
- **Re-importing a shop does not re-resolve proprietor cross-links** if the
  proprietor actor was imported later. Same future-sweep ticket.
- **Picker fetch parallelism** is `Promise.allSettled` — if dm-assistant is
  slow on one of the four list endpoints, the dialog still opens but that
  list's row will be empty with an inline error.

## If something fails

- Capture: screenshot of the chip, the Foundry console (`F12`) tab, the
  failing payload (Network tab → the `/foundry/<kind>/<slug>` request).
- File a bug in `kajecode/dm-assistant-foundry-bridge` with the v0.4.0 tag.
- If the failure looks like a contract-version mismatch (chip yellow), check
  dm-assistant's `/foundry/health` response — it should report `contract_version`
  ≥ 0.3.0. If it's lower, dm-assistant needs to be upgraded first.
