# Smoke Test — bridge v0.5.2

**Release:** [v0.5.2](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.5.2) · **Date:** 2026-05-15

Per-type `Item.system` fields for non-weapon items. Companion to
**dm-assistant v0.29.3** (LLM now emits per-spell entries). The two
ship together — without this v0.5.2 shape fix the now-emitted spell
items would land as unrenderable stubs.

Sections:

- A — Deploy both halves + version chip
- B — Re-extract a spellcaster NPC so the actions sidecar has spells
- C — Import: spell items render in the Spells tab
- D — Other item types (equipment / consumable / loot / tool) render
- E — Regression: weapons + feats from v0.5.0/v0.5.1 still work

## Prerequisites

- **dm-assistant ≥ 0.29.3** on the Pi (contract still 0.4.0 — no
  contract change in either release).
- bridge v0.5.2 installed in Foundry.
- Foundry v13 + dnd5e v5.x.
- A spellcaster NPC in a campaign (Elowen Tristane is the reference —
  a 4th-level druid).

---

## A. Deploy both halves

### dm-assistant (Pi)

```bash
ssh pi
cd dm-assistant
git fetch --tags && git checkout v0.29.3 && ./scripts/update-pi.sh
```

### bridge (Foundry)

Update the module from the v0.5.2 manifest URL:

```text
https://github.com/kajecode/dm-assistant-foundry-bridge/releases/latest/download/module.json
```

### A — Expected

- dm-assistant `/api/health` reports `0.29.3`.
- bridge chip green, version `v0.5.2`, contract `0.4.0` (unchanged).

## B. Re-extract the spellcaster

The actions sidecar for existing spellcaster NPCs was written by the
OLD prompt (spells folded into one feat). Re-extract with `--force`
so the v0.29.3 prompt runs:

```bash
ssh pi
cd dm-assistant
source .venv/bin/activate
python tools/extract_stats.py --campaign <id> --slug elowen-tristane --kind npc --force
```

(Drop `--slug` to re-extract the whole campaign.)

### B — Expected

Inspect the regenerated sidecar:

```bash
cat /home/piroot/share/data/campaigns/<id>/documents/dm/npc_elowen-tristane_dnd5e_actions.yaml
```

- Now contains **multiple `type: spell` entries** (Fire Bolt, Entangle,
  Goodberry, etc.) — one per spell named in Elowen's prose.
- Plus the weapon (Dagger) + any genuine feats (Nature's Ward).
- The old single "Spellcasting" catch-all feat may still appear (the
  trait itself is legitimately a feat) — that's fine. The point is
  the *individual spells* are now their own entries.

### B — Gate

If the sidecar still has zero `type: spell` entries after `--force`,
the dm-assistant v0.29.3 deploy didn't take, or the NPC's prose
doesn't actually signal spellcasting. Check the prose has a
Spellcasting trait / named spells.

## C. Import — spells render in the Spells tab

1. In Foundry, open the bridge picker → NPCs → Elowen Tristane →
   Import. (This re-imports; the drop-and-replace drift policy
   removes the old bridge-marked items first.)
2. Open the actor sheet → **Spells** tab.

### C — Expected

- The translated spells appear as **spell-type items** in the Spells
  tab, grouped under **Cantrips / level 0** (the v1 default — the
  actions schema doesn't carry spell level yet).
- Each spell item is named bare (no `(Elowen Tristane)` suffix —
  that decoration is weapon/feat only).
- Each spell carries `flags.dm-assistant-bridge.source =
  "dm-assistant"` (F12: `actor.items.find(i=>i.type==="spell").flags`).
- The spell's description shows the LLM's flavour text (or is empty
  if the LLM gave none).

### C — Known limit (not a bug)

All spells land at **level 0 (cantrip)** and **empty school**. That's
expected at v0.5.2 — the actions schema doesn't emit spell level. The
GM either:

1. Edits level/school on each spell's sheet, OR
2. Deletes the stub and drags the real spell from an SRD / DDB
   compendium, OR
3. Waits for **#32** (compendium-source resolution) which will
   auto-replace the stub with the fully-statted compendium spell.

The win in v0.5.2 is that the spells **exist as proper spell items**
that the GM can act on — previously they didn't exist at all.

## D. Other item types

If you have an NPC whose actions sidecar includes equipment /
consumable / loot / tool entries (e.g. "Studded Leather",
"Potion of Healing", "Thieves' Tools", "Gold Pouch"):

1. Import that NPC.
2. Open the actor sheet → **Inventory** tab.

### D — Expected

- Equipment / consumable / loot / tool items appear in Inventory,
  each as the correct dnd5e item type with:
  - `quantity: 1`, weight, price slots present (no "undefined"
    rendering)
  - Equipment shows an equip toggle
  - Names bare (no actor-name suffix)
- No blank / broken rows.

### D — If no such items exist

Skip — covered by the v0.5.2 unit tests
(`tests/dnd5e-items.test.ts::"per-type system fields (v0.5.1)"`).

## E. Regression — weapons + feats

1. On the same imported actor, check the Dagger (weapon) and any
   feats (Nature's Ward).

### E — Expected

- Dagger appears in Inventory, named `Dagger (Elowen Tristane)`.
- Clicking the Dagger's attack rolls `d20 + <to_hit>` then damage
  (the v0.5.1 `_id` fix means the activity actually fires).
- Feats appear in the Features tab, named `<feat> (Elowen Tristane)`.
- No regression from v0.5.0 / v0.5.1 behaviour.
