# Smoke Test — bridge v0.5.0

**Release:** [v0.5.0](https://github.com/kajecode/dm-assistant-foundry-bridge/releases/tag/v0.5.0) · **Date:** 2026-05-14

Embedded Items on imported actors (bridge#20). Closes the Objects→Items
v1 umbrella that began with dm-assistant #485 (the actions sidecar,
shipped in v0.28.0).

> ⚠ **Breaking for users on dm-assistant < 0.28.0.** `min-api-contract-version`
> bumps `0.3.0 → 0.4.0`. Upgrade the server first.

Sections:

- A — Install + version chip
- B — NPC import: actor gets embedded weapons + feats
- C — Click-to-roll: weapon attack actually rolls in Foundry
- D — Drift policy: re-import drops stale items, keeps user-authored ones
- E — Creature import: same translator covers the bestiary path
- F — Regression: payloads without `front_matter.actions` still import cleanly

## Prerequisites

- **dm-assistant ≥ 0.28.0** (contract 0.4.0). Older deployments will
  see the chip go yellow ("outdated") and import will be disabled.
- **Foundry v13** with **dnd5e v5.x** active system. (PF2e items
  ship in a separate translator — S12.)
- A campaign with at least one NPC that has been re-extracted post-v0.28.0
  (so it has an `_actions.yaml` sidecar). Either:
  - Generate a fresh NPC in the dm-assistant UI (the panel's `save`
    flow triggers actions extraction in the background), OR
  - Run `python tools/extract_stats.py --campaign <id>` on the Pi to
    backfill existing NPCs with the actions sidecar.

## A. Install + version chip

1. Foundry → **Game Settings → Manage Modules → Update Module** for
   DM Assistant Bridge (or install via manifest URL):

   ```text
   https://github.com/kajecode/dm-assistant-foundry-bridge/releases/latest/download/module.json
   ```

2. Reload the world.

### A — Expected

- Chip is **green**.
- Tooltip shows bridge `v0.5.0` and contract `0.4.0` (was `0.3.0`).
- No "outdated" yellow state.

### A — Gate

Stop here if the chip is yellow. The dm-assistant server is on an old
contract; upgrade to v0.28.0+ before continuing.

## B. NPC import — embedded items appear

Goal: import an NPC whose actions sidecar has at least one weapon
entry. Confirm the actor sheet shows the weapon AND a feat / trait.

1. Open the bridge picker, switch to **NPCs**, pick an NPC that you
   know has actions (use one generated post-v0.28.0; e.g. Aldric
   Harwick from a recent save).
2. Click **Import**.
3. Open the imported actor's sheet.

### B — Expected

- **Inventory** tab (dnd5e sheet's bag icon): weapon-type items
  appear, each named `<weapon name> (<actor name>)`. E.g.
  `Slam (Aldric Harwick)`.
- **Features** tab: feat-type items appear, also decorated with the
  actor name. E.g. `Pack Tactics (Aldric Harwick)`.
- **Spells** tab (if applicable): spell-type items appear (bare
  names, no actor suffix).
- Other item types (loot / equipment / consumable / tool) land in
  the dnd5e sheet's appropriate tabs with bare names.
- Each item carries a `flags.dm-assistant-bridge.source =
  "dm-assistant"` flag (visible via the item sheet's `</>` debug
  panel, or via the F12 console: `actor.items.contents[0].flags`).

## C. Click-to-roll — weapon attack works

Goal: confirm the synthesised `activities` block actually drives a
working attack roll.

1. On the imported actor's sheet, click a weapon-type item's
   attack-roll button (the sword icon, typically on the right edge
   of the item row).

### C — Expected

- Foundry rolls the attack as a `d20 + <to_hit>` (e.g. `d20 + 9` for
  Slam if `attack.to_hit: 9` was in the sidecar).
- The roll lands in the chat panel with a damage button.
- Clicking the damage button rolls the damage formula (`3d6+6` for
  Slam) and applies the right damage type (`bludgeoning`).

### C — Melee vs ranged classification

If you imported a creature with both a melee weapon AND a clearly-
ranged weapon (e.g. dragon's Slam + Fire Breath, or a hobgoblin's
longsword + longbow):

- Weapons with range < 30ft AND no `thr` / `amm` properties → roll as
  **melee weapon attack** (mwak) using Strength.
- Weapons with range ≥ 30ft OR `thr` / `amm` properties → roll as
  **ranged weapon attack** (rwak) using Dexterity.

GM can flip via the item sheet for edge cases the heuristic misses.

### C — Gate

If the weapon doesn't roll: check the item sheet's Activities tab.
Should have one entry of type "attack". If empty, the translator
didn't synthesise — file a bug with the source sidecar payload.

## D. Drift policy — re-import is drop-and-replace

Goal: prove that re-importing an NPC replaces the bridge-marked items
without disturbing user-authored items.

1. On the imported actor's sheet, **manually create** an item — e.g.
   add a new Feature called "GM Notes" with custom text. (Or drag a
   compendium item onto the actor.)
2. On the dm-assistant side, edit the NPC's prose or sidecar (or just
   re-save without changes — the modified_at timestamp ticks).
3. Re-import the same NPC via the bridge picker.

### D — Expected

- The dm-assistant-translated items are **regenerated**. The previously-
  imported `Slam (Aldric Harwick)` etc. are present (possibly with
  the same content) — they are technically new documents (different
  IDs) but the visible state is identical.
- Your hand-added "GM Notes" item **survives untouched**.

### D — Verify via flags

Open the F12 console:

```js
const a = game.actors.getName("Aldric Harwick");
a.items.contents.forEach(i => console.log(i.name, i.getFlag("dm-assistant-bridge", "source")));
```

You should see:
- `Slam (Aldric Harwick)` with source `"dm-assistant"`
- `Pack Tactics (Aldric Harwick)` with source `"dm-assistant"`
- `GM Notes` with source `undefined` (your hand-added item; no flag)

### D — Empty-actions edge case

If you re-import an NPC whose actions sidecar is now empty (e.g. you
deleted the sidecar manually), the bridge still runs the drop-and-
replace pass:

- All bridge-marked items are deleted.
- No new items are created.
- User-authored items still survive.

## E. Creature import

Goal: confirm the same translator covers the creature path. The
issue's gold standard is the Solyrian Keeper export — anything with
a rich actions list works.

1. In the picker, switch to **Creatures**, pick one with a sidecar
   (use the dm-assistant Pi to verify a `creature_<slug>_dnd5e_actions.yaml`
   exists for the slug).
2. Click **Import**.

### E — Expected

- Same shape as NPC import:
  - Weapons land in Inventory tab.
  - Feats/traits land in Features tab.
  - Names decorated with the creature name.
- Stats sidecar still merges (separate flow; was already working
  pre-v0.5.0).

## F. Regression — no actions sidecar still works

Older NPCs / Creatures (pre-v0.28.0) lack the `front_matter.actions`
field. The bridge should import biography + stats normally and skip
embedded items.

1. Pick an entity that was extracted under dm-assistant v0.25.0 or
   earlier (no actions sidecar on disk). Import.

### F — Expected

- Actor lands successfully.
- Biography populated.
- Stats block populated (if a stats sidecar exists).
- **No embedded items** appear — the Inventory/Features tabs are
  empty (or contain only items you manually added).
- No console errors; the picker's success toast fires normally.

### F — If you have only post-v0.28.0 entities

Skip this section — the empty-actions case is covered by unit tests
(`tests/buildActorData.test.ts::"returns empty items array when payload
lacks front_matter.actions"`).

## Known limits (not bugs)

- **`activities` is minimal.** A single attack-shape activity per
  weapon. Full dnd5e v5.x `activities` keyed-dict (multi-activity
  items, save mechanics, healing) is phase 2.
- **Ranged-vs-melee heuristic is conservative.** Defaults to melee
  unless range ≥ 30ft or `thr`/`amm` properties signal otherwise.
  GM can flip on the item sheet.
- **`object_slug` + `compendium_source`** on item entries are
  ignored at v1. They're reserved for dm-assistant #481 v2 (Objects
  Library cross-reference) and phase 2 (compendium-source
  resolution).
- **Spell items have description only.** No spell-system fields
  (level, school, components, etc.) — spell extraction is a
  separate ticket.

## If something fails

- Capture: F12 console output during import, the failing item's
  source sidecar payload (curl `/api/foundry/actor/<kind>/<slug>`
  on the Pi), screenshot of the actor sheet.
- File against `kajecode/dm-assistant-foundry-bridge` with the
  v0.5.0 tag.
